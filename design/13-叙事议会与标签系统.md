# 零下家园 · 小说世界模拟引擎 — 叙事议会与标签系统

> 定义引擎的第 4 层架构：五路 LLM 分诊叙事议会、全局标签系统、信息生命周期管理、长程叙事结构控制。
> 利用 DeepSeek V4 的 1M 上下文窗口，每 50 tick 一次全量叙事评审，成本 < $0.05/轮。

---

## 一、架构定位

```
前推引擎（第 1-3 层，Rust 数值引擎）
  │  每 tick 产出：事件流、角色行动、状态变化、关系图更新
  │  标注：每个事件携带标签元数据（类型、参与者、域、情感调性）
  ▼
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
                  叙事议会（第 4 层）
         每 50 tick 触发一轮（约每 5-10 章）
         1M 上下文：全量事件日志 + 角色状态 + 标签索引
         5 路并行 LLM 调用，各司其职
         输出：叙事指令（调整下阶段的参数）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  │
  ▼
参数写回 WorldState
  ├─ Storyteller 权重调整（事件类型偏好、节奏、紧张度目标）
  ├─ EmergentDetector 灵敏度（哪些碰撞更敏感）
  ├─ ProbabilityTree 分支概率偏移（某条路径权重±）
  ├─ Hook 优先级调整（哪些线索需要跟进）
  └─ 标签系统更新（激活/归档/降级）
```

## 二、标签系统（全局元数据层）

### 2.1 标签类型

```rust
/// 所有标签的联合类型
pub enum Tag {
    // ─── 事件标签 ───
    /// 事件类型标签（从 EventType 派生）
    Event(EventTag),
    /// 情感调性：该事件的情感基调
    Tone(EmotionalTone),
    /// 叙事功能：该事件在叙事中扮演的角色
    NarrativeFunction(NarrativeFunction),
    
    // ─── 角色标签 ───
    /// 角色当前弧线阶段
    ArcStage(String),
    /// 角色当前核心冲突
    CoreConflict(String),
    /// 角色当前焦点（角色正在关注什么）
    Focus(String),
    
    // ─── 剧情线索标签 ───
    /// 未解决的线索（伏笔）
    Thread(ThreadStatus),
    /// 线索类型
    ThreadType(ThreadType),
    
    // ─── 元标签 ───
    /// 信息质量
    Freshness(Freshness),
    /// 置信度（0~1）
    Confidence(f64),
}

pub enum EventTag {
    CharacterDevelopment,  // 角色成长
    RelationshipChange,    // 关系变化
    ResourceCrisis,        // 资源危机
    ExternalThreat,        // 外部威胁
    Discovery,            // 发现
    Conflict,             // 冲突
    Resolution,           // 解决
    WorldBuilding,        // 世界观展开
    Foreshadowing,        // 伏笔
    Callback,             // 呼应
    Pivot,               // 转折
    Climax,              // 高潮
}

pub enum EmotionalTone {
    Hope, Fear, Tension, Relief,
    Grief, Joy, Anger, Melancholy,
    Neutral,
}

pub enum NarrativeFunction {
    Setup,        // 设局
    Payoff,       // 收局
    Escalation,   // 升级
    Breathing,    // 喘息
    Transition,   // 过渡
    Reveal,       // 揭示
    Twist,        // 反转
}

pub enum ThreadStatus {
    Active,      // 活跃中——正在发展的线索
    Dormant,     // 休眠——暂时搁置，可被激活
    Resolved,    // 已解决
    Abandoned,   // 已废弃（不会复活）
    ForeshadowPing, // 已播种但未启动
}

pub enum ThreadType {
    Mystery,          // 谜团："超市地下室有什么？"
    Promise,          // 承诺："陈磊会带来他部队的朋友"
    Threat,           // 威胁："血手帮还在附近"
    Relationship,     // 关系："林霜和张远之间的信任"
    CharacterArc,     // 角色弧线："张远从普通人变成领袖"
    WorldSecret,      // 世界秘密："全球降温的真正原因"
}

pub enum Freshness {
    Current,    // 当前聚焦（显示在 Dashboard 上）
    Recent,     // 近期（30 tick 以内）
    Stale,      // 过期（30 tick 以上无更新）
    Archived,   // 归档（不再主动使用，可被伏笔侦探复活）
}
```

### 2.2 标签数据流

```
事件产生 → 自动打标签（基于 EventType + 上下文）
  ├─ 情感调性 → bhava 情绪引擎的输出决定
  ├─ 叙事功能 → 由事件类型 + 上下文环境推断
  └─ 线索状态 → 如果是新线索 → ForeshadowPing
                如果是已有线索的延续 → 更新对应 ThreadStatus
  
标签入库 → 全局标签索引
  ├─ Freshness: Current / Recent / Stale / Archived
  ├─ 每 tick: Recent→Stale（如果 30 tick 没更新）
  ├─ 每 tick: Stale→Archived（如果 60 tick 没更新）
  └─ 叙事议会可以手动调整 Freshness（复活伏笔）
  
标签影响 → 决策权重
  ├─ Storyteller: 如果某个 ThreadType 长期无更新 → 增加对应事件类型的概率
  ├─ EmergentDetector: Archived 标签不参与碰撞检测
  └─ 叙事议会: 扫描 Archived 标签，决定哪些值得复活
```

### 3.2 五路并行

```
每轮议会（5 路并行 LLM 调用）
  │
  ├─ ❶ 连续性审查官 Continuity Auditor
  │   输入：事件日志 + 角色标签 + 关系图摘要
  │   任务：检查角色一致性、设定连贯性、因果关系断裂
  │   输出：ContinuityReport（不一致列表 + 修复建议）
  │   Token: ~30K
  │
  ├─ ❷ 弧线规划师 Arc Architect
  │   输入：事件日志 + 标签索引 + 统计指标
  │   任务：评估当前叙事阶段，规划下阶段方向
  │   输出：ArcDirective（阶段、节奏、焦点、张力目标）
  │   Token: ~30K
  │
  ├─ ❸ 质量评审员 Quality Assessor
  │   输入：事件日志 + 叙事分数
  │   任务：找出最精彩和最乏味的部分
  │   输出：QualityReport（评分 + 建议）
  │   Token: ~30K
  │
  ├─ ❹ 伏笔侦探 Foreshadow Analyst
  │   输入：标签索引（重点关注 Archived/Dormant 线索）+ 事件日志
  │   任务：扫描被遗忘的线索，识别可复活的伏笔
  │   输出：ResurrectionProposals（复活建议 + 触发条件）
  │   Token: ~30K
  │
  └─ ❺ 执行总结 Executive Summarizer
      输入：前四路的输出
      任务：合并冲突、优先级排序、生成可执行的叙事指令
      输出：NarrativeDirective + 摘要
      Token: ~8K
```

## 四、信息生命周期管理

### 4.1 新鲜度流转

```
事件产生
  │
  ▼
Freshness::Current    ← 当前正在展示（Dashboard 高亮）
  │  事件被处理并推送到前端后
  ▼
Freshness::Recent     ← 30 tick 以内
  │  30 tick 无更新
  ▼
Freshness::Stale      ← 仍然可用但降权
  │  60 tick 无更新
  ▼
Freshness::Archived   ← 不再参与自动检测
  │  叙事议会可以手动复活
  ▼
（被伏笔侦探选中）→ Freshness::Current（复活）
```

### 4.2 自动维护

```rust
impl TagIndex {
    /// 每 tick 调用，维护标签新鲜度
    pub fn tick_maintenance(&mut self, current_tick: Tick) {
        // 1. Recent→Stale（30 tick 无更新）
        for thread in self.active_threads.values_mut() {
            if thread.freshness == Freshness::Recent 
                && current_tick - thread.last_updated > 30 {
                thread.freshness = Freshness::Stale;
            }
        }

        // 2. Stale→Archived（60 tick 无更新）
        self.active_threads.retain(|_, thread| {
            if thread.freshness == Freshness::Stale
                && current_tick - thread.last_updated > 60 {
                thread.freshness = Freshness::Archived;
                self.archived_threads.push(thread.clone());
                return false; // 从活跃索引移除
            }
            true
        });

        // 3. Current→Recent（已推送到前端）
        for thread in self.active_threads.values_mut() {
            if thread.freshness == Freshness::Current {
                thread.freshness = Freshness::Recent;
            }
        }
    }

    /// 叙事议会调用：复活归档线索
    pub fn resurrect(&mut self, thread_id: Uuid, new_freshness: Freshness) {
        if let Some(idx) = self.archived_threads.iter().position(|t| t.id == thread_id) {
            let mut thread = self.archived_threads.remove(idx);
            thread.freshness = new_freshness;
            thread.last_updated = current_tick;
            self.active_threads.entry(thread.thread_type).or_default().push(thread);
        }
    }
}
```

### 4.3 过时信息处理

事件日志不会删除，但标签可以降级。引擎始终保存完整历史，议会看到的上下文包含了所有历史（利用 1M 窗口），但**注意力权重自动向 Current/Recent 标签倾斜**。

## 五、DeepSeek V4 的 1M 上下文利用策略

### 5.1 Token 预算分配

每轮议会约需要 ~128K token（输入）+ ~8K（输出）。1M 上下文可以塞：

```
输入布局（每轮议会）：
  ├─ System Prompt（2K，固定）
  ├─ 自上次议会以来的事件流（~30K，约 50 tick）
  ├─ 全局标签索引摘要（~15K，所有活跃 + 休眠线索）
  ├─ 角色当前状态（~20K，全部角色）
  ├─ 关系图摘要（~5K）
  ├─ 统计指标（~2K）
  ├─ 上次议会指令 + 执行情况（~3K）
  ├─ 完整事件日志（不含已归档）（~30K）
  └─ 额外上下文窗口（~20K，可滑动到历史)）
```

**保留历史的方式**：不是每次都塞全部 1M。而是每轮议会自带自上次议会以来的增量日志 + 全部标签索引。如果需要回溯更早的历史，用关键词检索从事件日志中提取相关片段。

## 八、性能账本

| 组件 | 触发频率 | LLM 调用/轮 | Token/轮 | 等效 RPM | 成本/轮 |
|------|---------|------------|---------|---------|--------|
| 5 路议会 | 每 50 tick | 5 | ~128K | 0.1 | ~$0.03 |
| 标签维护 | 每 tick | 0 | 0 | 0 | $0 |
| 新鲜度更新 | 每 tick | 0 | 0 | 0 | $0 |

5 路议会并行调用（tokio::spawn），实际延迟 ≈ 最慢一路，约 5-15s（取决于 DeepSeek 当前负载）。但**它是异步的——不阻塞主循环**。主循环继续跑 tick，议会结果到了再写回。

## 九、效果预期

```
加入叙事议会前：
  ├─ 引擎能产出"合理的剧情素材"
  ├─ 但：角色会缓慢漂移、节奏不可控、伏笔没人收
  └─ 作者需要不断人工介入修正

加入叙事议会后：
  ├─ 引擎能产出"有自我审美的剧情素材"
  ├─ 每 50 tick 自我修正一次
  ├─ 伏笔自动检测 + 回收建议
  ├─ 节奏持续优化（慢了加速，快了喘息）
  └─ 作者从"编剧"降级为"主编"——审阅 + 确认，而不是从零写
```

议会的核心贡献不是"让引擎更聪明"，而是**让引擎有了自我修正能力**。一个会自己发现自己写崩了然后自己修的角色模拟引擎——这就是你说的"仔细打磨"的结果。
