### F4：锁策略——读写锁分片替代全局大锁

**问题**：`Arc<Mutex<SimulationEngine>>` 使所有操作串行化。LLM 调用期间（~3s）整个引擎被阻塞。

**方案**：拆分为 3 个读写锁 + 1 个无锁队列。

```rust
// backend/src/engine/mod.rs
pub struct SimulationEngine {
    /// 世界状态——读多写少，用 RwLock
    pub world: RwLock<WorldState>,
    /// 标签索引——叙事议会写入，主循环读取
    pub tag_index: RwLock<TagIndex>,
    /// 引擎配置——几乎只读，极少写入
    pub config: RwLock<EngineConfig>,
    /// 统计信息——原子操作，不需要锁
    pub stats: AtomicEngineStats,
    /// LLM 请求队列——无锁 mpsc channel
    pub llm_tx: mpsc::Sender<LLMRequest>,
}

// 主循环只持有 world 的读锁，LLM 调用在锁外进行
impl SimulationEngine {
    pub async fn detailed_tick(&self) {
        // 1. 读世界状态（读锁，多个 tick 可并发读）
        let world_snapshot = self.world.read().unwrap().clone();
        
        // 2. LLM 调用（不持有任何锁）
        let decisions = llm_gateway.batch_decide(&world_snapshot).await;
        
        // 3. 应用决策（写锁，短暂持有）
        let mut world = self.world.write().unwrap();
        for decision in decisions {
            world.apply_decision(decision);
        }
    }
}
```

**附：ECS 双重可变引用问题（来自审核 agent 发现）**

原 doc 02 的 `detailed_tick` 伪代码在同一作用域中对 `world.characters` 进行了两次 `.query::<&mut CharacterState>()` 和 `.query::<(&CharacterState, &mut CharacterActivity)>()`——这在 Rust 中借有检查器会拒绝。

**Rust 修复方案**：将查询阶段分拆为独立的块或用 scope 隔离：

```rust
pub async fn detailed_tick(&self) {
    // Phase 1: 只读取状态（共享引用）
    let idle_count;
    {
        let world = self.world.read().unwrap();
        idle_count = world.characters.query::<&CharacterState>().iter().count();
    }
    
    // Phase 2: LLM 调用（不持锁）
    let decisions = llm_gateway.batch_decide(&world_snapshot).await;
    
    // Phase 3: 分步更新（先更新状态，再更新活动）
    {
        let mut world = self.world.write().unwrap();
        // 只更新 CharacterState
        for decision in &decisions {
            world.update_states(decision);
        }
    }
    {
        let mut world = self.world.write().unwrap();
        // 只更新 CharacterActivity
        for decision in &decisions {
            world.update_activities(decision);
        }
    }
}
```

**关键规则**：hecs 的 `query::<&mut A>()` 和 `query::<&mut B>()` 分属两个写锁区间，永远不叠加借用。
```

**死锁预防**：
- 所有锁按固定顺序获取：`world → tag_index → config`（从不反向）
- 异步任务（如议会写回）中获取锁使用 `try_lock` + 重试，避免阻塞主循环
- LLM 网关回调中绝不获取引擎锁

### F5：优雅关闭 + 崩溃恢复

**问题**：没有 `POST /stop`、没有 SIGTERM handler、没有 WAL。关页面 = 丢数据。

**方案**：

```rust
// backend/src/main.rs
#[tokio::main]
async fn main() {
    // SIGTERM / SIGINT / Ctrl+C 处理
    let (shutdown_tx, mut shutdown_rx) = tokio::sync::mpsc::channel(1);
    
    tokio::spawn(async move {
        tokio::signal::unix::signal(SignalKind::terminate()).unwrap().recv().await;
        shutdown_tx.send(()).await.ok();
    });
    
    tokio::spawn(async move {
        tokio::signal::ctrl_c().await.unwrap();
        shutdown_tx.send(()).await.ok();
    });

    // 启动时：检查上次是否异常退出
    let crash_recovered = CrashRecovery::check().await;
    if crash_recovered {
        tracing::warn!("引擎上次异常退出，已从最新检查点恢复，可能丢失部分 tick");
    }

    // 启动 API 服务器
    let app = create_router(engine);
    let listener = tokio::net::TcpListener::bind("0.0.0.0:3000").await.unwrap();
    
    // 优雅关闭
    axum::serve(listener, app)
        .with_graceful_shutdown(async { shutdown_rx.recv().await; })
        .await
        .unwrap();
}
```

async fn stop_simulation(State(engine): State<Arc<SimulationEngine>>) -> Json<Value> {
    // 1. 暂停主循环
    engine.state.store(EngineState::Stopped, Ordering::SeqCst);
    
    // 2. 取消所有挂起的 LLM 调用
    engine.llm_cancel_token.cancel();
    
    // 3. 保存最终检查点
    engine.save_checkpoint("shutdown").await;
    
    // 4. 等待最多 5 秒让所有异步任务完成
    tokio::time::timeout(Duration::from_secs(5), async {

            tokio::time::sleep(Duration::from_millis(100)).await;
        }
    }).await.ok();
    
    // 5. 关闭 redb
    engine.db.close().await;
    
    Json(json!({"status": "ok", "saved": true}))
}

### 附：三个遗忘系统的关系与集成策略

审核 agent 发现 doc 06（角色级感知记忆）、doc 12（引擎级事件有效寿命）、doc 13（叙事标签新鲜度）各有独立的遗忘机制。

**它们不是冲突，是三个不同层级的遗忘**：

```
层级 1：角色级记忆（doc 06）
  作用对象：单个角色的记忆检索
  衰减函数：Ebbinghaus R(t)=exp(-t/S)
  人格影响：是（高神经质衰减慢）
  M1 范围：全部（核心角色系统）
  
层级 2：引擎级事件寿命（doc 12）
  作用对象：事件队列中级联的触发能力
  衰减函数：ForgettingCurve.activation()
  人格影响：否（全局统一）
  M1 范围：基础（事件不过期，M2 再加衰减）
  
层级 3：叙事标签新鲜度（doc 13）
  作用对象：线索/标签的活跃状态
  衰减函数：固定 tick 阈值（30/60）
  人格影响：否（全局统一）
  M1 范围：不需要（M4 议会才用）
```

**M1 实现规则**：
- 层级 1（角色记忆）全部实现——核心角色行为依赖它
- 层级 2（事件寿命）只实现基础版本——事件永久 `activation=1`，不衰减，M2 再加 `ForgettingCurve`
- 层级 3（标签新鲜度）暂不实现——M4 叙事议会才需要

这样三个系统在 M1 不冲突。等到 M2/M4 接入时，每条事件的生命周期以层级 2 为准，层级 1 和层级 3 分别引用同一时间源。

        ))

// 使用示例：每个 tick 执行的关键路径都有 trace
pub async fn detailed_tick(engine: &SimulationEngine) {
    let tick_span = span!(Level::INFO, "detailed_tick", tick = engine.current_tick());
    let _guard = tick_span.enter();
    
    info!("开始 Detailed tick");
    // ...
    
    for char_id in &need_llm {
        let char_span = span!(Level::DEBUG, "llm_decision",
            char = %char_id,

        let result = llm_gateway.call_llm(perceive_prompt).await;
